## 1. 需求评审（开发视角）

### 1.1 评审结论

| 维度 | 结论 |
|------|------|
| **与 PRD 对齐** | 以 **智能问答 Agent** 为一等公民，知识库为 **RAG 数据源**；与「ima.copilot 类」产品形态 **一致**。 |
| **可开发性** | MVP 技术路径成熟：FastAPI + SSE + 向量库 + 兼容 LLM API；**难点在引用质量与解析**，需预留评测与调参。 |
| **闭环** | **问 → 检 → 答 → 引用** 已落表；**反馈驱动优化** 依赖 P1；**删除与向量 GC、embedding 变更重索引** 需在实现/详设收口（见 **§9**）。 |

### 1.2 需求澄清清单（建议编码前收口）

| ID | 条目 | 说明 |
|----|------|------|
| D1 | **单库 vs 多库同屏** | MVP 仅 **单 KB 绑定会话**；多库合并检索放 P1。 |
| D2 | **引用展示粒度** | 仅片段高亮，还是 **页码/章节**（PDF 需解析元数据）？ |
| D3 | **无检索结果** | 固定拒答模板 vs 允许模型「泛泛而谈」（建议 **拒答或明确说明未命中**）。 |
| D4 | **Embedding 模型** | 与向量库维度锁死；更换模型需 **全量重嵌入**。 |
| D5 | **对象存储** | 上传文件存本地磁盘（开发）还是 MinIO/OSS（生产）？ |
| D6 | **模型提供方** | **用户端落库**（`user_id` 归属）vs 系统环境变量；**密钥加密**（AES/KMS） |

## 2. 技术评审

### 2.1 可行性

| 能力 | 评估 |
|------|------|
| SSE 流式对话 | 成熟 |
| RAG（向量 + Prompt） | 成熟；中文效果依赖 **切块与 embedding** |
| MySQL 存元数据 + Qdrant/小规模向量 | 与当前技能栈匹配 |
| PDF 解析 | 可用 PyMuPDF/pdfminer；复杂排版有损 |

### 2.2 架构原则

1. **密钥不外露**：LLM Key、Embedding Key **仅服务端**。
2. **检索在服务端**：前端只传 `conversation_id`、`message`、`kb_id`；**禁止**前端传整段知识库文本给模型绕过检索（**MVP 未定义「无 KB 纯对话」**，若未来增加须单独开场景并改门禁策略）。
3. **引用可解析**：assistant 消息存 **`citation: [{chunk_id, doc_id, excerpt}]`**，便于审计与点击跳转。
4. **降级**：向量服务挂 → 提示维护；LLM 挂 → 固定错误页；**检索空** → 模板拒答。

### 2.3 风险与对策

| 风险 | 对策 |
|------|------|
| 幻觉 | 系统提示 + 空检索不答 + 强制引用格式 |
| 切块切断语义 | overlap、按标题分段、调 chunk_size |
| 成本 | 限流、摘要历史、小模型做路由（P2） |

## 3. 系统总体架构

### 3.1 逻辑架构

```
┌─────────────────────────────────────────────────────────────┐
│  Vue3 客户端（对话页 + 知识库管理页 + **模型设置页** + 登录）      │
└────────────────────────────┬────────────────────────────────┘
                             │ HTTPS / SSE
┌────────────────────────────▼────────────────────────────────┐
│  FastAPI BFF                                                 │
│  - Auth（JWT）                                                │
│  - 会话与消息 CRUD                                             │
│  - POST /v1/conversations/{id}/messages (SSE) → AgentService  │
└────────────────────────────┬────────────────────────────────┘
                             │
        ┌────────────────────┼────────────────────┐
        ▼                    ▼                    ▼
┌───────────────┐   ┌─────────────────┐   ┌─────────────┐
│ AgentService  │   │ IngestionService │   │ Admin       │
│ RAG + LLM     │   │ 解析/切块/嵌入    │   │ 配置/审计    │
└───────┬───────┘   └────────┬──────────┘   └──────┬──────┘
        │                    │                     │
        ▼                    ▼                     ▼
┌───────────────────────────────────────────────────────────────┐
│ MySQL 8：用户、会话、消息、KB、文档元数据、chunk 元数据、审计      │
│ Redis：队列、限流、会话热数据（可选）                            │
│ Qdrant（或备选）：向量 + payload（kb_id, doc_id, chunk_id）      │
│ Object Storage：原始文件                                       │
└───────────────────────────────────────────────────────────────┘
        │
        ▼
┌───────────────┐
│ Worker        │ 解析、embed、upsert 向量、更新 MySQL 状态
└───────────────┘
```

### 3.2 推荐技术栈

| 层级 | 选型 | 备注 |
|------|------|------|
| 前端 | **Vue 3 + TS + Vite + Element Plus + Pinia + Vue Router** | |
| 聊天 UI | 自研消息列表 + **markdown 渲染**（**markdown-it** + **highlight.js**）；引用用 **el-card** 列表 | |
| SSE | `fetch` + `ReadableStream` 或 `fetch-event-stream` 类库 | |
| 后端 | **FastAPI** + **httpx**（调 LLM） | |
| Agent 编排 | **LangChain**（Retriever + chain）或 **LangGraph**；MVP 可用 **单函数 RAG** 降低复杂度 | |
| ORM | **SQLAlchemy 2** + **Alembic** | 驱动 **PyMySQL** |
| 向量 | **Qdrant**（Docker）推荐；或 **numpy 本地 Top-K**（仅原型/小数据） | |
| 队列 | **ARQ** + Redis | |
| LLM 接入 | **统一 OpenAI 兼容客户端**（`base_url` + `api_key` + `model` 动态注入） | 多厂商见 **§3.4** |

### 3.3 数据库分工（MySQL 为主）

| 存什么 | 放哪里 |
|--------|--------|
| 用户、会话、消息文本、KB、文档记录、索引状态 | **MySQL** |
| 向量 | **Qdrant**（或 MySQL 存 chunk 文本 + 向量在 Qdrant 用 id 对齐） |
| 原始文件 | **对象存储** 或本地 `uploads/` |

**不必**为 RAG 先学 PostgreSQL；**MySQL + 向量库** 是常见组合。

### 3.4 模型提供方与统一调度（对应 PRD §5.4）

| 设计要点 | 说明 |
|----------|------|
| **抽象** | `LLMProvider`（api_base、api_key 密文、provider_type、extra_headers JSON、**`user_id` 可空**）+ `ModelProfile`（model_id、用途 chat/embedding、is_default、enabled） |
| **MVP（强制）** | **MySQL 落库** + **`/v1/me/providers`** + **`/v1/me/model-readiness`**；**生产环境聊天与入库嵌入** 仅以 **用户落库配置** 为准 |
| **聊天门禁** | **`is_model_ready(user_id)`**：存在启用中的 **默认 chat** 与 **默认 embedding**（各至少一条）；**未就绪** → 禁止 `POST /conversations` 与 `POST .../messages`，返回 **`MODEL_NOT_READY`** |
| **解析顺序（就绪后）** | **`ModelResolver`**：① 会话 `chat_model_id`（P1）→ ② 用户默认 chat / embedding；**不再**用环境变量 **自动替代** 生产路径（开发可用 `ALLOW_CHAT_WITHOUT_MODEL_SETUP` 仅影响网关） |
| **P2** | **可选** `/v1/admin/providers` 组织级 |
| **嵌入切换** | 用户更换默认 embedding 时，提示并触发 **该用户知识库** 重索引 |
| **实现提示** | Python **OpenAI SDK** `base_url` 动态注入；密钥 **按 user_id** 隔离查询、**加密落库** |

## 4. 前端技术实现方案

### 4.1 页面划分

| 路由 | 功能 |
|------|------|
| `/chat` | 会话列表 + 主聊天区 + **当前 KB 选择器**；**仅当 `model-readiness.ready===true` 可进入**（否则守卫重定向 **`/settings/models`**） |
| `/knowledge` | 知识库列表、新建、进入文档列表 |
| `/knowledge/:id` | 文档上传、状态、删除、重试索引 |
| `/login` | 登录 |
| `/settings/models`（**MVP 必选**） | **模型与密钥**：提供方/模型 CRUD、**连通性测试**、Key 脱敏、默认 chat/embedding；**首次登录默认落地页**（未就绪时） |
| `/admin/models`（P2，可选） | 组织级策略或审计视图；**不替代** 用户端自配 |

### 4.2 路由守卫与登录跳转（MVP）

| 要点 | 说明 |
|------|------|
| **守卫** | `beforeEach` 中：若目标为 `/chat` 且 `ready=false` → **`replace('/settings/models')`**，并可选 `ElMessage.warning` |
| **白名单** | `/login`、`/settings/models`、`/knowledge`（仅浏览；**上传** 仍受 **MODEL_NOT_READY** 约束）、静态资源 **不拦截** |
| **就绪拉取** | 登录后或 `providers` 变更后刷新 **Pinia `modelReadiness`**；**保存成功** 后重新请求 readiness，**`true` 再跳转 `/chat`** |
| **开发绕过** | `import.meta.env.VITE_ALLOW_CHAT_WITHOUT_MODEL_SETUP === 'true'` 时跳过守卫（仅本地） |

### 4.3 对话模块要点

| 模块 | 要点 |
|------|------|
| **流式** | 后端 SSE：`event: token` / `event: citation` / `event: done` / `event: error` |
| **引用** | 消息对象带 `citations[]`；点击可打开 **文档预览（P1）** 或仅展示片段 |
| **选择 KB** | 新建会话前 `el-select` 绑定 `kb_id`；无 KB 时引导先上传 |
| **选择对话模型（P1）** | `el-select` 绑定 `chat_model_id`（**P1**；MVP 仅用 **用户在设置页保存的默认对话模型**） |
| **停止** | `AbortController` 取消 fetch |

### 4.4 知识库管理要点

| 模块 | 要点 |
|------|------|
| 上传 | `el-upload` + 分片（大文件 P1）；展示 **processing / ready / failed** |
| 列表 | 文档名、大小、状态、操作 |

## 5. 后端技术实现方案

### 5.1 包结构建议

```
app/
  api/           # auth, conversations, chat, knowledge_bases, documents, ingest
  services/
    rag.py       # retrieve → build_prompt → stream_llm
    agent.py     # 可选：多步；MVP 单轮 RAG + 历史
  workers/       # parse, chunk, embed, upsert_qdrant
  integrations/  # openai_compatible_client（动态 base_url/key）
  models/
  llm_registry/     # MVP: DB + ModelResolver；生产不以 env 顶替就绪态
```

### 5.2 核心 API

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/v1/auth/login` | 登录 |
| GET | `/v1/me/model-readiness`（**MVP**） | `{ ready: bool, missing?: ["chat","embedding"] }` |
| GET/POST/PATCH/DELETE | `/v1/me/providers`（**MVP**） | **当前用户** 的模型提供方 CRUD |
| GET/POST/PATCH | `/v1/me/providers/{id}/models`（**MVP**） | 模型实例 CRUD、设默认 chat/embedding |
| POST | `/v1/me/providers/{id}/probe`（**MVP**） | 连通性测试 |
| GET/POST/PATCH | `/v1/admin/providers`（P2，可选） | 组织级提供方（若启用多租户） |
| GET/POST | `/v1/knowledge-bases` | KB 列表/创建 |
| POST | `/v1/knowledge-bases/{id}/documents` | 上传（multipart） |
| GET | `/v1/knowledge-bases/{id}/documents` | 文档列表 |
| POST | `/v1/conversations` | 创建会话；**`is_model_ready` 失败 → 412 + `MODEL_NOT_READY`** |
| POST | `/v1/conversations/{id}/messages` | **SSE 流式**；同上 |

### 5.3 RAG 管道（单次问答）

0. **`is_model_ready(user_id)`**，否则 **直接返回** `MODEL_NOT_READY`（不进入检索）。  
1. 校验用户对该 `kb_id` 的权限。
2. 用户消息 **embedding**（使用 **当前默认 embedding 模型** 或会话策略）→ Qdrant `filter: kb_id` → Top-K。
3. 组装 messages：`system`（规则 + 引用格式）+ `检索片段` + 历史若干轮 + `user`。
4. 经 **ModelResolver** 解析 **对话模型**（会话 `chat_model_id`（P1）或 **用户默认 chat**）→ 调 LLM **stream**；流式结束后 **持久化** assistant 消息 + **citations**。
5. 若 K=0：不调用 LLM 或调用 **短模板模型** 输出「未找到」。

### 5.4 文档入库（异步）

1. 上传接口若 **模型未就绪** → **409/412** + `MODEL_NOT_READY`，**不入队**；或 **queued** 但 Worker 首步校验失败并标记 `failed`（二选一，推荐 **上传前拦截**）。  
2. 保存文件 → 写 `documents` 表 `status=queued`。  
3. Worker：解析文本 → 切块 → 每块 embedding → upsert Qdrant → 更新 `status=ready`。

### 5.5 安全

| 项 | 实现 |
|----|------|
| 鉴权 | JWT；`kb_id` 与 `user_id` 关联表校验；**`/v1/me/*` 强制 `provider.user_id == 当前用户`** |
| 提示注入 | 用户内容置于明确边界；系统提示不可覆盖 |
| 限流 | Redis |
| **模型门禁** | 所有需 LLM/Embedding 的写接口 **先 `is_model_ready`**；错误体与前端 **`MODEL_NOT_READY`** 对齐 |

### 8.2 待收口清单（实现前建议补详设）

| ID | 说明 |
|----|------|
| T1 | **删除文档/KB/Provider**：MySQL 删除后 **异步删除 Qdrant points**（按 `doc_id`/`kb_id` payload 过滤），避免幽灵向量 |
| T2 | **Worker 上下文**：入库 Worker 必须带 **`user_id` → 解析默认 embedding**，与主链路 ModelResolver 规则一致 |
| T3 | **Qdrant 与维度**：若 embedding 模型维度变更，需 **新 collection 或重建任务**；与 PRD G5 对应 |
| T4 | **`chunks.qdrant_point_id`**：建议与 Qdrant point id 一一对应，便于删除与对账 |
| T5 | **幂等**：`POST .../messages` 重试时 **message_id** 或 **client_request_id** 防重复计费与重复插入 |
| T6 | **SSE 断开**：客户端中断时 **是否停止上游 LLM**（节省 Token）作为非功能优化项 |

### 8.3 已知简化假设

| 假设 | 风险 |
|------|------|
| 单租户、单部署 | 多租户时需拆分 Qdrant collection 或 `tenant_id` payload |
| Qdrant 单机 | 需备份与恢复策略（运维向，非 MVP 功能文档范围） |
