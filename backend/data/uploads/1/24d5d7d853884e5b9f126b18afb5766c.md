# 前端打包后部署到另一台服务器 - 配置说明

本文说明如何配置 Jenkins，使前端项目构建产物（`dist/`）通过 **rsync + SSH** 自动部署到**另一台服务器**（与 Jenkins 不在同一台机器）。

---

## 一、整体流程

1. Jenkins 构建完成后，使用 **rsync** 通过 **SSH** 将 `dist/` 同步到目标服务器的指定目录。
2. Jenkins 使用「SSH 私钥」凭据连接目标服务器，无需输入密码。
3. 目标服务器需提前创建好目录、配置好 Nginx 等（本文仅说明 Jenkins 与 SSH 相关配置）。

---

## 二、私钥从哪里来？能用密码吗？

### 私钥不是从目标服务器“获取”的

- **私钥**是你在 **Jenkins 所在机器**（或你本机）上**自己生成**的一对密钥里的“私钥”那一份。
- **目标服务器**上只放**公钥**（放在 `~/.ssh/authorized_keys` 里），用来验证“拿私钥的人”可以登录。
- 流程是：你在 Jenkins 服务器上执行 `ssh-keygen` 生成一对密钥 → 把**公钥**拷到目标服务器 → 把**私钥**内容填到 Jenkins 凭据里。目标服务器不会、也不应该把它的私钥给你。

### 能用密码连接吗？

可以，但不推荐（密码写在凭据里、易泄露，且需在 Jenkins 机器上装 `sshpass`）。若必须用密码：

1. 在 Jenkins 添加类型为 **Username with password** 的凭据，用户名填目标机 SSH 用户，密码填该用户密码，记下凭据 ID（如 `deploy-password`）。
2. Jenkins 所在机器安装：`sudo apt install sshpass`（或 `yum install sshpass`）。
3. 流水线部署阶段改为使用该密码凭据并配合 `sshpass` 执行 rsync（见下节「使用密码的流水线示例」）。

**推荐**仍使用 SSH 密钥方式，更安全且无需在目标机开放密码登录。

---

## 三、目标服务器上的准备

### 3.1 创建部署目录

在**要部署前端的那台服务器**上执行（将用户与路径按需修改）：

```bash
# 创建目录（与 Jenkins 里 DEPLOY_SSH_PATH 一致）
sudo mkdir -p /var/www/html/digital-platform-front

# 假设用用户 deploy 通过 SSH 接收文件，把属主改为 deploy
sudo chown -R deploy:deploy /var/www/html/digital-platform-front

# 若 Nginx 用 www-data，可改为 deploy:www-data 并 chmod 775
```

### 3.2 配置 SSH 免密登录（Jenkins 用私钥连接）

在 **Jenkins 所在服务器**上，为 Jenkins 使用的用户（通常是 `jenkins`）生成一对密钥，或使用已有私钥。下面以新生成为例：

```bash
# 以 jenkins 用户执行（或 root 执行后把私钥交给 Jenkins 凭据）
sudo su - jenkins
ssh-keygen -t ed25519 -C "jenkins-deploy" -f ~/.ssh/jenkins_deploy -N ""
```

将**公钥**写入**目标服务器**上部署用户的 `authorized_keys`：

```bash
# 在 Jenkins 服务器上查看公钥
cat ~/.ssh/jenkins_deploy.pub

# 在目标服务器上，以 deploy 用户执行
mkdir -p ~/.ssh
chmod 700 ~/.ssh
echo "这里粘贴上面输出的公钥内容" >> ~/.ssh/authorized_keys
chmod 600 ~/.ssh/authorized_keys
```

在 Jenkins 服务器上测试免密登录（可选）：

- **若当前已是 root 或其他用户**（未登录为 jenkins），可执行：
  ```bash
  sudo su - jenkins -c 'ssh -i ~/.ssh/jenkins_deploy -o StrictHostKeyChecking=no deploy@目标服务器IP "echo ok"'
  ```
  此时提示输入的是**你当前用户的密码**（有 sudo 权限的账号），不是 jenkins 的密码。
- **若已经以 jenkins 用户登录**（提示符为 `jenkins@主机名:~$`），**不要用 sudo**，直接执行：
  ```bash
  ssh -i ~/.ssh/jenkins_deploy -o StrictHostKeyChecking=no deploy@目标服务器IP "echo ok"
  ```
  能输出 `ok` 即表示 SSH 配置正确。

#### 完整操作命令示例（目标服务器 IP：192.168.1.245，用户：root）

**步骤 1：在 Jenkins 服务器上生成密钥**

```bash
sudo su - jenkins
ssh-keygen -t ed25519 -C "jenkins-deploy" -f ~/.ssh/jenkins_deploy -N ""
exit
```

**步骤 2：在 Jenkins 服务器上查看公钥**

```bash
sudo su - jenkins -c 'cat ~/.ssh/jenkins_deploy.pub'
```

复制输出的整行公钥，用于步骤 3。

**步骤 3：在目标服务器 192.168.1.245 上写入公钥**

先登录目标服务器（用户按实际，如 root 或 deploy）：

```bash
ssh root@192.168.1.245
```

在目标服务器上执行（将 `这里粘贴第2步的整行公钥` 替换为步骤 2 复制的公钥）：

```bash
mkdir -p ~/.ssh
chmod 700 ~/.ssh
echo '这里粘贴第2步的整行公钥' >> ~/.ssh/authorized_keys
chmod 600 ~/.ssh/authorized_keys
exit
```

**步骤 4：在 Jenkins 服务器上测试免密登录**

若当前是 root 等用户：

```bash
sudo su - jenkins -c 'ssh -i ~/.ssh/jenkins_deploy -o StrictHostKeyChecking=no root@192.168.1.245 "echo ok"'
```

若当前已是以 jenkins 登录：

```bash
ssh -i ~/.ssh/jenkins_deploy -o StrictHostKeyChecking=no root@192.168.1.245 "echo ok"
```

输出 `ok` 即表示配置成功。流水线里 `DEPLOY_SSH_USER` 需与这里一致（如 `root`）。

## 四、Jenkins 凭据配置

1. 登录 Jenkins → **系统管理** → **凭据** → **添加凭据**。
2. 类型选择 **SSH Username with private key**。
3. 填写：
   - **ID**：填固定英文 ID，如 `deploy-ssh-key`（与流水线里 `DEPLOY_SSH_KEY_CREDENTIAL_ID` 一致）。
   - **用户名**：目标服务器上的 SSH 用户，如 `deploy`（与 `DEPLOY_SSH_USER` 一致）。
   - **Private Key**：选 **Enter directly**，把 Jenkins 服务器上 `~/.ssh/jenkins_deploy`（私钥）的**完整内容**粘贴进去（含 `-----BEGIN ... KEY-----` 和 `-----END ... KEY-----`）。
4. 保存。

#### DEPLOY_SSH_KEY_CREDENTIAL_ID 如何获取？

**不是从别处“查出来”的**，而是你在添加上述 SSH 凭据时，在 **ID** 那一栏**自己填的英文标识**（如 `deploy-ssh-key`）。流水线里的 `DEPLOY_SSH_KEY_CREDENTIAL_ID` 填这个 ID 即可。若未填 ID，Jenkins 会自动生成一长串，可在「凭据」列表中查看该凭据旁显示的 ID。

## 五、流水线环境变量说明

在 **Jenkinsfile** 或 **Pipeline 脚本**的 `environment` 中配置以下变量，即可在「部署」阶段把打包结果推到另一台服务器。

| 变量                           | 说明                                        | 示例                                   |
| ------------------------------ | ------------------------------------------- | -------------------------------------- |
| `DEPLOY_TARGET`                | 部署目标：`local` 本机，`remote` 远程服务器 | `remote`                               |
| `DEPLOY_SSH_HOST`              | 目标服务器 IP 或域名                        | `192.168.1.100` 或 `front.example.com` |
| `DEPLOY_SSH_USER`              | 目标服务器 SSH 登录用户                     | `deploy`                               |
| `DEPLOY_SSH_PATH`              | 目标服务器上的目录（前端静态资源根目录）    | `/var/www/html/digital-platform-front` |
| `DEPLOY_SSH_KEY_CREDENTIAL_ID` | Jenkins 中 SSH 私钥凭据的 ID                | `deploy-ssh-key`                       |

**部署到本机**时：将 `DEPLOY_TARGET` 设为 `local`，并配置 `DEPLOY_PATH`（本机路径），不填 `DEPLOY_SSH_HOST` 即可。

**部署到远程**时：将 `DEPLOY_TARGET` 设为 `remote`，并正确填写上表各变量。

---

## 六、Jenkinsfile 中的配置示例

在 **Jenkinsfile** 里（或任务 → 配置 → 流水线 → 环境变量）设置：

```groovy
environment {
    BUILD_MODE = 'production'
    DEPLOY_TARGET = 'remote'
    DEPLOY_PATH = '/var/www/oa-front'
    DEPLOY_SSH_HOST = '192.168.1.100'
    DEPLOY_SSH_USER = 'deploy'
    DEPLOY_SSH_PATH = '/var/www/html/digital-platform-front'
    DEPLOY_SSH_KEY_CREDENTIAL_ID = 'deploy-ssh-key'
}
```

若使用 **Pipeline script from SCM**（仓库内 Jenkinsfile），可在 Jenkins 任务配置里通过「注入环境变量」或直接改 Jenkinsfile 中上述变量。

## 七、使用密码的流水线示例（不推荐，仅作备选）

若使用「Username with password」凭据（ID 如 `deploy-password`），且 Jenkins 机器已安装 `sshpass`，可将部署阶段改为：

```groovy
withCredentials([usernamePassword(credentialsId: 'deploy-password', usernameVariable: 'SSH_USER', passwordVariable: 'SSH_PASS')]) {
    sh """
        sshpass -p '\$SSH_PASS' rsync -avz --delete -e "ssh -o StrictHostKeyChecking=no -o UserKnownHostsFile=/dev/null" \
            dist/ \$SSH_USER@${env.DEPLOY_SSH_HOST}:${env.DEPLOY_SSH_PATH}/
    """
}
```

需将 `deploy-password` 改为你创建的密码凭据 ID，并确保 `DEPLOY_SSH_HOST`、`DEPLOY_SSH_PATH` 已配置。

## 八、依赖与行为说明

- **rsync**：Jenkins 所在机器需已安装 rsync（Ubuntu 一般已带）。流水线中使用 `rsync -avz --delete`，会保持目标目录与 `dist/` 一致（多余文件会被删掉）。
- **SSH**：使用 `-o StrictHostKeyChecking=no -o UserKnownHostsFile=/dev/null` 避免首次连接交互，适合受控内网；若需严格校验主机密钥，可去掉这两项并在目标机预先加入 known_hosts。
- **权限**：目标目录应对 `DEPLOY_SSH_USER` 可写；若 Nginx 以另一用户读文件，需自行设置属主/属组（如 `deploy:www-data` 与 775）。

## 九、常见问题

| 现象                                       | 可能原因                              | 处理                                                                                                                          |
| ------------------------------------------ | ------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------- |
| 部署阶段报 CredentialId 找不到             | Jenkins 里未添加或 ID 填错            | 检查「凭据」中 SSH 私钥的 ID 与 `DEPLOY_SSH_KEY_CREDENTIAL_ID` 一致                                                           |
| Permission denied (publickey)              | 目标机未加入公钥或用户错              | 检查目标机 `~/.ssh/authorized_keys` 与 `DEPLOY_SSH_USER`                                                                      |
| rsync: command not found                   | Jenkins 节点未安装 rsync              | 在 Jenkins 所在机器执行 `sudo apt install -y rsync`（或 `yum install rsync`），然后重新执行构建                               |
| 部署后页面 404                             | Nginx 根路径或 try_files 未指向该目录 | 检查 Nginx root 或 alias 指向 `DEPLOY_SSH_PATH`，并配置 try_files                                                             |
| sudo su - jenkins 时提示「jenkins 的密码」 | 当前用户已是 jenkins，又执行了 sudo   | jenkins 系统用户通常无可用密码；若已登录为 jenkins，不要用 sudo，直接执行 ssh/ssh-keygen 等命令                               |
| 不知道 jenkins 用户的密码                  | 系统设计如此                          | jenkins 用户一般无登录密码；`sudo su - jenkins` 时输入的是**你当前登录用户的密码**（有 sudo 权限的账号），不是 jenkins 的密码 |

### sudo su - jenkins 是什么意思？

- **sudo**：用管理员权限执行后面的命令（会要求输入**当前用户**的密码）。
- **su - jenkins**：切换为系统用户 **jenkins**，并进入其家目录（`-` 表示登录式切换）。
- 合起来：用当前用户权限「变成」jenkins 用户执行操作。Jenkins 服务是以 jenkins 用户运行的，在该用户下生成的 SSH 密钥才能被流水线正常使用。

## 十、指定分支 push 后自动构建（无 Webhook 时）

若 Git 仓库**没有 Webhook**，无法在 push 时主动通知 Jenkins，可以用 **轮询 SCM（Poll SCM）** 实现「指定分支有提交就自动构建」。

### 1. 任务只构建指定分支

- 若流水线里用 `environment { GIT_BRANCH = 'test' }` 且 checkout 使用该变量，则该任务只会拉取并构建该分支。
- 若使用「Pipeline script from SCM」，在「源码管理 → Git」中把**分支**写成 `*/test` 或 `*/main`（与你要监听的分支一致）。

建议：一个任务对应一个分支（如 test 任务只监听 test，prod 任务只监听 main）。

### 2. 启用轮询 SCM

1. 打开该 Jenkins 任务 → **配置**。
2. 在 **「构建触发器」** 中勾选 **「轮询 SCM（Poll SCM）」**。
3. 在调度框中填写 Cron 表达式，例如：
   - **每 5 分钟检查一次**（推荐）：`H/5 * * * *`
   - **每 1 分钟检查一次**：`* * * * *`
4. 保存。

Jenkins 会按该周期检查远程仓库的**该任务配置的分支**是否有新提交；有则自动触发构建，无则跳过。无需 Webhook，仅存在轮询间隔带来的延迟。

---

按上述步骤配置后，每次流水线执行到「部署」阶段，会把当前构建的 `dist/` 同步到另一台服务器的指定目录，实现前端打包后自动部署到远程服务器。
