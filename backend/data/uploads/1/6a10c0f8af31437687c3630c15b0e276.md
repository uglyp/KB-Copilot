# Jenkins 配置常见问题与解决

本文档整理在本项目（full-process-digital-platform）接入 Jenkins 自动构建与部署过程中遇到的问题及解决方案，便于后续排查与新人参考。

---

## 一、安装与端口

### 1.1 自定义端口不生效，访问不到 Jenkins

**现象**：已修改为 38080（或其它端口），`systemctl status jenkins` 显示 running，但 `ss -tlnp | grep 38080` 无输出，浏览器访问 `http://服务器IP:38080` 失败。

**原因**：`apt install jenkins` 安装完成时会自动用**默认 8080** 启动服务；脚本或手动只改了配置并执行了 `systemctl start jenkins`，而服务已在运行，`start` 不会重启，新端口未生效。

**解决**：改端口后必须**重启**服务：

```bash
sudo systemctl daemon-reload
sudo systemctl restart jenkins
sudo ss -tlnp | grep java   # 确认新端口在监听
```

详见项目内 `docs/jenkins-install-guide.md` 中「3.3 修改端口」「3.4 已安装后修改端口」。

---

### 1.2 是否必须使用 8080 端口？

**答**：不必。可通过 systemd 覆盖配置使用任意未被占用的端口（如 38080、9080），参见安装文档中的「修改端口」章节。

---

### 1.3 服务器已有 Java 21，是否还要装 Java 17？

**答**：不需要。Jenkins 支持 Java 11、17、21，使用现有 Java 21 即可。

---

### 1.4 安装前如何检查磁盘与内存？

安装脚本已包含安装前检查：要求 `/var` 所在分区至少 2GB 可用，建议可用内存 ≥ 512MB；不足会提示或退出。详见 `scripts/install-jenkins-ubuntu.sh`。

## 二、Git 与凭据

### 2.1 克隆仓库报鉴权失败（CredentialId 找不到 / 鉴权失败）

**现象**：

- `Warning: CredentialId "git-credentials-id" could not be found.`
- `fatal: 'https://...' 鉴权失败`

**原因**：流水线中使用了占位符凭据 ID，或未在 Jenkins 中配置对应 Git 凭据。

**解决**：

1. 登录 Jenkins → **系统管理** → **凭据** → **添加凭据**。
2. 类型选 **Username with password**，填写 Git 用户名和密码（或 Personal Access Token），**ID** 填固定英文 ID（如 `savaboon-git`）。
3. 在流水线或任务配置中，将 `GIT_CREDENTIALS_ID` / 凭据下拉框改为刚创建的 ID，**不要**选「- none -」。

## 三、Node 与构建环境

### 3.1 构建时报 `node: not found`

**现象**：流水线「安装依赖」阶段执行 `node -v` 时报 `node: not found`。

**原因**：执行构建的 Jenkins 节点（本机或 agent）上未安装 Node.js，或未加入 PATH。

**解决（任选其一）**：

- **在 Jenkins 服务器上安装 Node 和 pnpm**（推荐）：

  ```bash
  curl -fsSL https://deb.nodesource.com/setup_20.x | sudo -E bash -
  sudo apt-get install -y nodejs
  sudo npm install -g pnpm
  ```

  确认 `sudo -u jenkins node -v` 能执行；若不能，在 Jenkins **系统管理 → 系统配置 → 全局属性** 中为 PATH 补充 Node 所在目录。

- **使用 Jenkins NodeJS 插件**：安装 NodeJS Plugin，在「全局工具配置」中新增 Node 安装（如 NodeJS-20），在流水线中声明 `tools { nodejs 'NodeJS-20' }`。

---

### 3.2 pnpm install 报 lockfile 与 overrides 不一致

**现象**：  
`ERR_PNPM_LOCKFILE_CONFIG_MISMATCH  Cannot proceed with the frozen installation. The current "overrides" configuration doesn't match the value found in the lockfile`

**原因**：`package.json` 中 overrides 等配置已改，但仓库里的 `pnpm-lock.yaml` 未同步更新。

**解决**：

- **临时**：流水线中改为执行 `pnpm install`（去掉 `--frozen-lockfile`），构建可先通过。
- **长期**：在本地执行 `pnpm install`，提交并推送更新后的 `pnpm-lock.yaml`，流水线再恢复使用 `pnpm install --frozen-lockfile` 以保证依赖一致。

## 四、部署与权限

### 4.1 部署时 cp 报 Permission denied

**现象**：  
`cp: cannot create regular file '/var/www/html/digital-platform-front/...': Permission denied`

**原因**：Jenkins 以 `jenkins` 用户运行，对部署目录无写权限（目录属主多为 root 或 www-data）。

**解决**：在部署目标服务器上为部署目录授权，例如：

```bash
sudo mkdir -p /var/www/html/digital-platform-front
sudo chown -R jenkins:www-data /var/www/html/digital-platform-front
sudo chmod -R 775 /var/www/html/digital-platform-front
```

将 `www-data` 改为实际运行 Nginx/Web 服务的用户（如 `nginx`）。

## 五、流水线与缓存

### 5.1 安装依赖逻辑：何时执行 pnpm install？

当前流水线逻辑：

- 根据 `package.json` + `pnpm-lock.yaml` 的哈希生成缓存键。
- 若该哈希对应的缓存中已有 `node_modules`，则**复用缓存**，不执行 install。
- 否则执行 `pnpm install` 并写入缓存。
- 仅保留最近 5 个缓存目录，避免长期占用大量磁盘；使用 `find -delete` 清理，不依赖 `rm`。

因此：**没有可用 node_modules 或依赖发生变化时会执行 install，未变化时跳过**。

---

### 5.2 缓存会占满磁盘吗？

不会无限增长。流水线在每次「安装依赖」后会对 `node-cache-${JOB_NAME}` 做清理，只保留**最近 5 个**缓存目录（按修改时间），其余用 `find ... -delete` 删除；并排除 `__stale__` 目录（若采用「mv 到 **stale**」的备用方案）。

---

### 5.3 若环境禁用了 rm，清理缓存会失败吗？

不会。清理逻辑已改为使用 **`find ./$d -delete`**，不依赖 `rm`。若连 `find -delete` 也被限制，可改为将待删目录 **mv** 到 `${CACHE_ROOT}/__stale__/`，由管理员或定时任务另行清理（见 Jenkinsfile 内注释）。

## 六、创建任务与插件

### 6.1 自动拉取 Git、打包、部署应选哪种任务类型？

推荐选择 **「流水线 (Pipeline)”**。在「新建 Item」中输入任务名称后，选择「流水线」，即可在 Pipeline 脚本或 Jenkinsfile 中定义：拉取代码 → 安装依赖 → 代码检查 → 构建 → 归档 → 部署。

---

### 6.2 首次配置时「安装推荐的插件」如何操作？

在「自定义 Jenkins」页面，直接点击左侧 **「安装推荐的插件」**，等待安装完成即可。若需自选插件，可点「选择插件来安装」。

| 现象                      | 可能原因                     | 处理方向                                               |
| ------------------------- | ---------------------------- | ------------------------------------------------------ |
| 改端口后仍访问不到        | 未重启 Jenkins               | `systemctl daemon-reload && systemctl restart jenkins` |
| Git 鉴权失败              | 未配置或填错凭据 ID          | 系统管理 → 凭据 → 添加并选用正确 ID                    |
| node: not found           | 节点未装 Node 或不在 PATH    | 安装 Node+pnpm 或配置 NodeJS 插件                      |
| pnpm lockfile 不匹配      | overrides 与 lockfile 不一致 | 先去掉 --frozen-lockfile；长期需提交新 lockfile        |
| 部署 cp Permission denied | 部署目录权限不足             | chown jenkins:www-data 部署目录并 chmod 775            |
| 缓存占满磁盘              | 旧版无清理逻辑               | 使用当前 Jenkinsfile（保留 5 个缓存 + find -delete）   |
